下面这套方案可以直接交给 Codex 分阶段开发。核心目标是：

> 公司上传资质、人员、业绩、投标偏好 → 系统自动拉取招投标公告 → 解析公告和附件 → 与公司能力库对比 → 自动判断“可投 / 不建议 / 需人工确认”。

# 一、系统总体架构

```text
招投标网站
  ↓
采集器 Crawler
  ↓
公告列表 / 公告详情 / 附件下载
  ↓
正文解析 / PDF解析 / Word解析
  ↓
结构化入库
  ↓
规则引擎硬判断
  ↓
AI语义分析
  ↓
综合评分
  ↓
后台展示 / 每日推送 / 人工确认
```

建议技术栈：

```text
前端：React + TypeScript + Shadcn UI
后端：Node.js + TypeScript + NestJS / Express
采集：Playwright + Axios + Cheerio
数据库：PostgreSQL
任务队列：BullMQ + Redis
文件存储：本地 MinIO / 阿里云 OSS
AI模型：通义千问 / Kimi / DeepSeek / OpenAI
文档解析：pdf-parse / mammoth / libreoffice 转换
```

## 3. 采集策略

优先顺序：

```text
1. 先找接口 JSON
2. 接口可用：Axios 直接请求
3. 接口不可用：Cheerio 解析 HTML
4. 页面 JS 渲染：Playwright 打开页面后解析
```

不要一开始就全用 Playwright，性能差。

---

## 4. 去重逻辑

每条公告生成 hash：

```ts
hash = sha256(sourceSite + url + title)
```

入库时判断：

```text
url 已存在 → 不重复入库
url 不存在但 content_hash 相同 → 视为重复
标题相同 + 发布时间相同 → 可能重复，标记 duplicate_suspected
```

# 七、自动判断逻辑

建议采用 **硬规则 + 评分 + AI总结**。

---

## 1. 硬规则过滤

直接判定不建议：

```text
1. 城市不在可投范围
2. 项目类型明显不匹配
3. 投标截止时间已过
4. 剩余时间少于公司配置的最小准备天数
5. 明确要求资质，公司没有
6. 明确要求人员，公司没有
7. 明确不接受联合体，但公司条件不足
8. 金额超过公司最大承接范围
9. 公告属于监理/设计/咨询，但公司只投施工
```

---

## 2. 匹配评分

总分 100：

```text
地区匹配：10分
项目类型匹配：15分
资质匹配：25分
人员匹配：20分
业绩匹配：15分
金额匹配：10分
时间充足：5分
```

评分结果：

```text
85 - 100：推荐参与
70 - 84：可关注
50 - 69：需人工确认
低于 50：不建议参与
硬性不满足：明确不满足
```

---

## 3. 资质匹配示例

公告要求：

```text
市政公用工程施工总承包三级及以上
```

公司资质：

```text
市政公用工程施工总承包二级
```

判断：

```text
满足
```

等级比较规则：

```text
特级 > 一级 > 二级 > 三级
甲级 > 乙级 > 丙级
```

---

## 4. 人员匹配示例

公告要求：

```text
项目负责人须具备市政公用工程专业二级及以上注册建造师
```

公司人员：

```text
张三：二级建造师，市政公用工程，有效
```

判断：

```text
满足
```

---

## 5. 业绩匹配示例

公告要求：

```text
近三年承担过单项合同金额 500 万元以上的市政道路工程
```

公司业绩：

```text
2024年 淮安市政道路改造项目 680万元
```

判断：

```text
基本满足
```

如果 AI 不确定，则标记：

```text
需人工确认
```

```text
每个站点请求间隔 2-5 秒
每天固定时间采集
失败自动重试 3 次
```

---

## 2. 保留原文

所有公告必须保存：

```text
原文链接
HTML原文
纯文本
附件
AI解析结果
人工确认结果
```

这样后续出错可以回溯。

---

## 3. AI结果必须可追溯

每个判断最好显示依据：

```text
公告原文要求：市政公用工程施工总承包三级及以上
公司资质：市政公用工程施工总承包二级
判断：满足
```

不要只显示“AI认为可以投”。

---

## 4. 一定要有人工确认

招投标不能完全自动决定，系统结论应该是：

```text
辅助判断
风险提示
资料清单
匹配推荐
```

最终仍由投标负责人确认。

---

# 十四、最终开发顺序建议

按这个顺序让 Codex 开发：

```text
第1步：搭建后端项目和数据库
第2步：建立 TenderNotice / CompanyProfile 等核心表
第3步：实现南京站公告列表采集
第4步：实现南京站公告详情采集
第5步：实现公告入库和去重
第6步：实现公司资质、人员、业绩手工录入
第7步：实现基础规则匹配
第8步：实现项目分析结果表
第9步：实现招标公告后台列表
第10步：实现项目详情页
第11步：接入 AI 提取公告要求
第12步：接入 AI 总结分析结果
第13步：扩展淮安、镇江、连云港采集器
第14步：实现附件下载和解析
第15步：实现每日定时任务
第16步：实现企业微信/钉钉推送
第17步：实现上传公司文件并自动识别
第18步：实现导出分析报告
```

---

一句话总结：

**这个系统完全能做，第一期核心不是 AI，而是“公告采集稳定 + 公司能力库准确 + 规则匹配可追溯”。AI 用来提取复杂文字和生成分析报告，不能完全替代硬规则判断。**
